인터넷에서 자원의 **위치와 상관없이 이름으로 영구적으로 식별**하는 URI의 하위 개념입니다.

## 구조

```
urn:isbn:0451450523
───  ────  ──────────
urn  NID     NSS
     (네임스페이스) (실제 식별자)
```

|요소|설명|예시|
|---|---|---|
|**urn**|URN임을 나타내는 고정 scheme|`urn`|
|**NID**|네임스페이스 식별자|`isbn`, `uuid`, `ietf`|
|**NSS**|실제 자원 식별값|`0451450523`|

## 실사용 예시

|예시|설명|
|---|---|
|`urn:isbn:0451450523`|책의 ISBN 번호|
|`urn:uuid:550e8400-e29b`|UUID로 자원 식별|
|`urn:ietf:rfc:2141`|IETF RFC 문서|

## URL vs URN

||URL|URN|
|---|---|---|
|**식별 방법**|위치(주소)|이름|
|**자원 이동 시**|링크 깨짐 💀|영구 유효 ✅|
|**사용 빈도**|매우 흔함|거의 안 씀|
|**예시**|`https://example.com/book`|`urn:isbn:0451450523`|

## URN이 잘 안 쓰이는 이유

```
URN으로 자원을 찾으려면
→ URN을 실제 위치(URL)로 변환하는 리졸버(Resolver) 필요
→ 인프라 구축 복잡
→ 현실적으로 URL 사용이 훨씬 간편
```

> URN은 자원이 어디로 이동해도 **영구적으로 식별 가능**하다는 장점이 있지만, 실제 접근을 위한 인프라 복잡성으로 인해 **실무에서는 거의 URL을 사용**합니다.
