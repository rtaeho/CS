TCP에서 데이터를 주고받기 전에 클라이언트와 서버가 서로 연결을 확인하는 3단계 과정입니다.

## 왜 필요한가?

- 양쪽 모두 통신 준비가 되었는지 확인
- 서로의 초기 순서 번호(Sequence Number) 교환
- 신뢰성 있는 연결을 위한 사전 작업

## 3단계 과정

```
클라이언트                    서버
    │                          │
    │─── 1. SYN (seq=100) ────▶│
    │    "연결해도 될까?"        │
    │                          │
    │◀── 2. SYN+ACK ───────────│
    │    (seq=200, ack=101)    │
    │    "응, 나도 연결할게"     │
    │                          │
    │─── 3. ACK (ack=201) ────▶│
    │    "확인, 시작하자"        │
    │                          │
    └──────── 연결 완료 ────────┘
```

## 각 단계 설명

|단계|플래그|의미|
|---|---|---|
|1|**SYN**|클라이언트가 연결 요청 + 내 시작 번호 알려줌|
|2|**SYN+ACK**|서버가 요청 수락 + 내 시작 번호 알려줌|
|3|**ACK**|클라이언트가 확인 → 연결 완료|

## 왜 3번이어야 하나?

- **2번이면 부족**: 클라이언트는 서버가 응답한 걸 알지만, 서버는 클라이언트가 받았는지 모름
- **3번이면 충분**: 양쪽 모두 상대방이 준비됐음을 확인

## 비유

전화 연결과 비슷합니다.

1. "여보세요?" (SYN)
2. "네, 말씀하세요" (SYN+ACK)
3. "아, 잘 들리네요" (ACK) → 대화 시작

## 면접 포인트

- SYN Flood 공격: 1단계만 대량으로 보내서 서버 자원 고갈시키는 DDoS 공격
- 이를 막기 위해 SYN Cookie 같은 방어 기법 사용
