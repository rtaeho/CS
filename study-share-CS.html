프로세스 vs 스레드
OS · 운영체제
프로세스는 OS로부터 자원을 할당받아 실행 중인 프로그램의 인스턴스이고,
스레드는 프로세스 내부의 실행 흐름 단위입니다.

OS · 운영체제

발생 원인
  • 타임 슬라이스 만료
  • I/O 대기 (파일, 네트워크)
  • 하드웨어 인터럽트
  • 높은 우선순위 프로세스 등장
발생 비용
  • PCB 저장 / 복원 비용
  • CPU 캐시 초기화 (Cache Miss)
  • 전환 중 CPU 유휴 상태
  • 프로세스 > 스레드 (비용 차이)
💡 스레드 과다 생성 시 컨텍스트 스위칭이 폭증해 오히려 성능이 저하됩니다.
스레드 풀(Thread Pool)로 CPU 코어 수에 맞게 제한하는 것이 핵심입니다.
데드락 (Deadlock)
OS · 운영체제
둘 이상의 프로세스가 서로 상대방이 점유한 자원을 기다리며 무한히 대기하는 상태입니다.
발생 조건 (4가지 모두 충족 시 발생)
1
상호 배제자원은 한 번에 하나의 프로세스만 사용 가능
2
점유 대기자원을 점유한 채 다른 자원을 기다림
3
비선점다른 프로세스의 자원을 강제로 빼앗을 수 없음
4
순환 대기프로세스들이 원형으로 서로의 자원을 대기
💡 Java에서는 락 획득 순서를 통일하면 순환 대기 조건이 깨져 데드락을 방지할 수 있습니다.
3-Way Handshake
Network · TCP
TCP 연결 수립 시 클라이언트와 서버가 SYN → SYN-ACK → ACK 세 단계를 교환해 신뢰성 있는 양방향 통신을 준비하는 과정입니다.
① SYN
클라이언트 → 서버
"연결하고 싶어"
클라이언트 ISN 전달
② SYN-ACK
서버 → 클라이언트
"좋아, 내 번호는~"
서버 ISN + 확인
③ ACK
클라이언트 → 서버
"확인 완료!"
연결 수립 (ESTABLISHED)
왜 3번인가?
ISN이 랜덤인 이유
양방향 4가지 (송·수신 × 2)를 최소 3단계에 담을 수 있는 이론적 최솟값입니다. 2-way면 유령 연결(Half-Open)이 생길 수 있습니다.
이전 연결의 지연 패킷이 새 연결에 섞이는 것 방지 + TCP 세션 하이재킹 공격 방어
CORS
Network · 브라우저 보안
다른 출처(Origin)의 리소스 접근을 제어하는 브라우저의 보안 메커니즘입니다.
Origin = Scheme + Host + Port (셋 중 하나라도 다르면 Cross-Origin)
SOP (동일 출처 정책)

  • 서버가 허용한 출처만 통과
  • Preflight(OPTIONS)로 사전 확인
  • Max-Age 캐싱으로 최적화 가능

트랜잭션 & ACID
Database
트랜잭션은 전부 성공하거나 전부 실패해야 하는 하나의 작업 단위입니다.
ex. 계좌 이체 — A에서 빠지고 B에 안 들어가면 10만원 증발!

설명

실행 전후 데이터 무결성 항상 유지

인덱스 (Index)
Database
데이터를 빠르게 찾기 위한 자료구조입니다. 책의 색인(목차)처럼, 전체를 뒤지지 않고 바로 위치를 찾아갑니다.
인덱스 없을 때
  • Full Table Scan 발생
  • 100만 건 → 100만 번 비교
  • WHERE 조건 느림
  • ORDER BY 느림
인덱스 있을 때 (B-Tree)
  • 트리 탐색으로 검색
  • 100만 건 → 약 20번 비교
  • O(log N) 시간 복잡도
  • 정렬·범위 검색에 유리
인덱스 쓰면 좋은 경우
인덱스 쓰면 나쁜 경우
• WHERE·JOIN에 자주 사용하는 컬럼
• 카디널리티가 높은 컬럼 (값 종류 많음)
• 조회 비율이 쓰기보다 높은 경우
• 값 종류가 적은 컬럼 (성별, 상태)
• INSERT/UPDATE/DELETE가 잦은 경우
• 테이블 데이터가 매우 적은 경우
💡 인덱스는 조회는 빠르게 하지만, INSERT/UPDATE/DELETE 시 인덱스도 함께 업데이트되므로 쓰기 성능이 저하됩니다.
클로저 (Closure)
JavaScript · FE
함수가 자신이 생성된 환경(변수)을 기억하는 것입니다.
외부 함수가 끝나도 내부 함수가 참조하는 변수가 사라지지 않고 유지됩니다.
function createCounter() { let count = 0; // 외부 함수 변수 — 원래는 종료 시 사라져야 함 return { increment: () => ++count, // count를 기억! getCount: () => count }; } const counter = createCounter(); // 함수 실행 끝남 counter.increment(); // 1 → count가 살아있음! counter.increment(); // 2 counter.getCount(); // 2 → 외부에서 count 직접 접근 불가
발생 조건
  • 내부 함수가 외부 변수를 참조
  • 내부 함수가 외부로 반환됨
  • 렉시컬 환경이 유지됨
활용 사례
  • private 변수 구현
  • React useState 내부 원리
  • 이벤트 핸들러 상태 유지
  • 커링(Currying) 함수
